⏱️ 3. Relógio Lógico de Lamport
02:45 - 04:30 (1 min 45s)
Ação Visual (Tela) Mostrem o arquivo internal/clock/lamport.go no editor de código, ou a seção 03 do relatório.
Pessoa A
"Entrando agora no coração do projeto: O Algoritmo de Lamport. Em sistemas distribuídos, não existe um 'horário global'. Então como sabemos a ordem dos eventos? Lamport resolve isso usando a relação de causalidade."
Pessoa B
"Para isso, escrevemos a struct LamportClock. A regra é rígida e implementamos em 3 funções cruciais. Primeiro, qualquer evento puramente local dentro do nó aciona a função Tick(), que apenas soma 1 ao relógio."
Pessoa A
"Segundo: se o nó vai enviar uma mensagem pela rede TCP, ele chama a função SendTick(). Ele soma 1 ao próprio relógio e anexa esse timestamp no JSON da mensagem."
Pessoa B
"E o terceiro é o mais importante: Ao receber uma mensagem, nós chamamos a ReceiveTick(ts). Ela compara o relógio local do nó com o relógio que chegou na mensagem, pega o maior valor entre os dois, e soma 1. Isso garante que a linha do tempo do recebedor sempre seja empurrada para frente caso o remetente estivesse num 'futuro lógico'."
🔒 4. Ricart-Agrawala (Exclusão Mútua)
04:30 - 06:45 (2 min 15s)
Ação Visual (Tela) Mudem para o arquivo internal/mutex/ricart_agrawala.go (focar na função RequestCS e HandleRequest) ou no fluxo de estados (RELEASED -> WANTED -> HELD) do relatório HTML.
Pessoa A
"Com a linha do tempo garantida por Lamport, enfrentamos o problema de exclusão mútua. Como garantir que dois nós não acessem um mesmo banco de dados ou arquivo simultaneamente, sem usar um servidor central para gerenciar filas? Usamos o algoritmo de Ricart-Agrawala."
Pessoa B
"A lógica de estado do nó se baseia em permissões e possui 3 fases: RELEASED (livre), WANTED (quer acessar) e HELD (está na seção crítica). Quando queremos acessar o recurso, mudamos para WANTED e enviamos um REQUEST via broadcast (para todos os outros 4 nós)."
Pessoa A
"O verdadeiro brilho da nossa implementação está em quem recebe a mensagem. Se o nó recebedor está RELEASED, ele responde com um REPLY na mesma hora. Se ele já está na seção crítica (HELD), ele trava a resposta e enfileira a requisição numa lista em memória."
Pessoa B
"E o cenário mais crítico: e se os dois nós estiverem em WANTED, brigando pela seção crítica ao mesmo tempo? Neste caso de colisão, nós comparamos os timestamps do pedido. Quem pediu mais cedo logicamente ganha a permissão. Se os timestamps forem bizarramente idênticos, usamos o ID numérico do nó como desempate matemático. O nó só altera seu estado para HELD após receber os exatos 4 REPLYs esperados. É um controle de concorrência perfeito e totalmente descentralizado."

06:45 - 08:15 (1 min 30s)
Ação Visual (Tela) Mostrem a lógica de timeout no internal/election/bully.go ou o esquema na arquitetura.
Pessoa A
"Para completar, precisamos de tolerância a falhas. Implementamos o Algoritmo Bully para reeleger um líder automaticamente se o nó principal cair."
Pessoa B
"A arquitetura funciona assim: nós mantemos um gatilho constante via `time.NewTicker` no Go para emitir pulsos (Heartbeats). Se a tentativa de envio via TCP retornar erro, sabemos instantaneamente que o líder caiu. O nó que detectou isso convoca uma eleição."
Pessoa A
"O Bully opera pela lei do mais forte. O nosso nó só envia a mensagem ELECTION para processos com ID estritamente maior que o dele. Se um nó maior estiver vivo, ele obrigatoriamente responde com um OK. Esse OK interrompe a tentativa do nó menor, e o nó maior assume o controle, iniciando sua própria eleição."
Pessoa B
"Quando chegamos num nó onde ninguém maior responde (por exemplo, o Nó 4, caso o Nó 5 tenha caído), o timeout de rede aciona, e ele se autodeclara vencedor, mandando uma mensagem de COORDINATOR para todos os nós vivos abaixo dele se reorganizarem."
