01. Visão Geral do Projeto

Tres algoritmos fundamentais de sistemas distribuídos implementados com comunicação real via sockets TCP.

⏲

Relógio Lógico de Lamport

Ordenacao causal de eventos entre processos distribuídos sem relógio global. Cada processo mantem um contador lógico que avança com eventos locais e troca de mensagens.

Lamport, 1978
🔒

Exclusão Mútua Distribuida

Algoritmo de Ricart-Agrawala para garantir acesso exclusivo a recurso compartilhado. Baseado em permissão: processo pede autorizacao a todos os outros.

Ricart & Agrawala, 1981
👑

Eleição de Líder (Bully)

Eleição automatica quando o líder falha. O processo com maior ID entre os vivos assume a liderança. Deteccao de falha via heartbeat com timeout.

Garcia-Molina, 1982

O Problema Fundamental

Em um sistema distribuído, cada processo tem seu próprio relógio (dessincronizado), não possui visão global do estado dos outros processos, e so pode obter informacoes trocando mensagens com latencia variavel. Isso gera três problemas classicos:

ProblemaSolução ImplementadaComplexidade
"Em que ordem os eventos aconteceram?"Relógio Lógico de LamportO(1) por evento
"Como garantir acesso exclusivo ao recurso?"Ricart-Agrawala2(N-1) msgs por entrada
"Quem coordena quando o líder cai?"Algoritmo BullyO(N²) pior caso

03. Relógio Lógico de Lamport

Ordenacao causal de eventos sem relógio global — base para os demais algoritmos.

Conceito Teorico

Leslie Lamport (1978) definiu a relacao "happened-before" (→):

  1. Dentro de um processo: Se o evento a ocorre antes de b em P, entao a → b
  2. Entre processos: Se a e o envio de uma msg por P1 e b e o recebimento por P2, entao a → b
  3. Transitividade: Se a → b e b → c, entao a → c

Regras do Algoritmo

Regra 1: Evento local → relógio = relógio + 1
Regra 2: Envio de mensagem → relógio = relógio + 1, anexa relógio na msg
Regra 3: Recebimento → relógio = max(relógio, timestamp_recebido) + 1
Exemplo Visual — Relógio de Lamport
Processo P1          Processo P2          Processo P3
(relógio=0)          (relógio=0)          (relógio=0)

[evento local]
relógio=1
     |
     |---msg(ts=1)-->
                     relógio=max(0,1)+1=2
                     [evento local]
                     relógio=3
                          |
                          |---msg(ts=3)-->
                                          relógio=max(0,3)+1=4
                                          [evento local]
                                          relógio=5

Garantia: Se a → b, entao L(a) < L(b). O timestamp lógico respeita a causalidade.

Implementação — internal/clock/lamport.go

lamport.go
type LamportClock struct {
    mu   sync.Mutex
    time int
}

// Tick incrementa o relógio para evento local
func (lc *LamportClock) Tick() int {
    lc.mu.Lock()
    defer lc.mu.Unlock()
    lc.time++
    return lc.time
}

// SendTick incrementa e retorna timestamp para a mensagem
func (lc *LamportClock) SendTick() int {
    lc.mu.Lock()
    defer lc.mu.Unlock()
    lc.time++
    return lc.time
}

// ReceiveTick: max(local, recebido) + 1
func (lc *LamportClock) ReceiveTick(receivedTimestamp int) int {
    lc.mu.Lock()
    defer lc.mu.Unlock()
    if receivedTimestamp > lc.time {
        lc.time = receivedTimestamp
    }
    lc.time++
    return lc.time
}
Por que sync.Mutex? Multiplas goroutines (servidor TCP, heartbeat, eleição) acessam o relógio concorrentemente. Sem o mutex, haveria race condition. Go detecta isso com go test -race.

04. Exclusão Mútua — Ricart-Agrawala

Acesso exclusivo a recurso compartilhado sem coordenador central — baseado em permissão.

Conceito Teorico

O algoritmo de Ricart-Agrawala e uma otimizacao do algoritmo original de Lamport para exclusão mútua. Usa timestamps logicos para desempatar pedidos simultaneos.

Estados de cada processo

EstadoRELEASED
→
EstadoWANTED
→
EstadoHELD
→
EstadoRELEASED

Protocolo

Para ENTRAR: Seta estado=WANTED, envia REQUEST(ts, id) para todos, aguarda N-1 REPLYs, seta estado=HELD.

Ao RECEBER REQUEST:
• Se RELEASED → envia REPLY imediatamente
• Se HELD → enfileira o REQUEST
• Se WANTED → compara timestamps (menor ts tem prioridade; desempata por menor ID)

Para SAIR: Seta estado=RELEASED, envia REPLY para todos os enfileirados.
Diagrama de Sequencia — Dois processos concorrentes (mesmo timestamp)
P1 (id=1)              P2 (id=2)              P3 (id=3)

[Quer entrar, ts=3]    [Quer entrar, ts=3]
REQ(3,1)------>        REQ(3,2)------>
REQ(3,1)----------------------------->
               <-------REQ(3,2)
                       REQ(3,2)------>

P3 esta RELEASED:
  Responde REPLY para P1 e P2

P1 recebe REQ(3,2):
  ts iguais, id1(1) < id2(2) => P1 tem prioridade
  P1 ENFILEIRA o request de P2

P2 recebe REQ(3,1):
  ts iguais, id1(1) < id2(2) => P1 tem prioridade
  P2 envia REPLY para P1

P1: recebeu REPLY de P2 + P3 => ENTRA na SC
P2: recebeu REPLY de P3, espera REPLY de P1

[P1 sai da SC] => envia REPLY para P2 (enfileirado)
P2: recebeu REPLY de P1 => ENTRA na SC

Implementação — internal/mutex/ricart_agrawala.go

ricart_agrawala.go — RequestCS (solicitar seção crítica)
func (ra *RicartAgrawala) RequestCS() {
    ra.mu.Lock()
    ra.state = Wanted
    ra.requestTimestamp = ra.clock.SendTick()
    ts := ra.requestTimestamp
    ra.mu.Unlock()

    // Broadcast REQUEST para todos os outros nos
    ra.transport.Broadcast(network.Message{
        Type:      "REQUEST",
        SenderID:  ra.nodeID,
        Timestamp: ts,
    })

    // Aguarda N-1 replies via channel (bloqueia)
    needed := ra.totalNodes - 1
    for i := 0; i < needed; i++ {
        <-ra.replyChan   // bloqueia até receber sinal
    }

    ra.mu.Lock()
    ra.state = Held
    ra.mu.Unlock()
}
ricart_agrawala.go — handleRequest (decidir REPLY ou enfileirar)
func (ra *RicartAgrawala) handleRequest(msg network.Message) {
    ra.clock.ReceiveTick(msg.Timestamp)
    ra.mu.Lock()
    defer ra.mu.Unlock()

    shouldDefer := false

    switch ra.state {
    case Held:
        shouldDefer = true   // estou na SC, enfileira
    case Wanted:
        if msg.Timestamp > ra.requestTimestamp {
            shouldDefer = true // meu pedido e mais antigo
        } else if msg.Timestamp == ra.requestTimestamp &&
                  msg.SenderID > ra.nodeID {
            shouldDefer = true // mesmo ts, meu ID menor = prioridade
        }
    case Released:
        // não quero entrar, REPLY imediato
    }

    if shouldDefer {
        ra.deferredReplies = append(ra.deferredReplies, msg.SenderID)
    } else {
        go ra.transport.Send(msg.SenderID, network.Message{
            Type: "REPLY", SenderID: ra.nodeID,
            Timestamp: ra.clock.SendTick(),
        })
    }
}

Propriedades Garantidas

PropriedadeDescrição
SafetyNo máximo 1 processo na seção crítica por vez
LivenessTodo processo que pede eventualmente entra
OrdenacaoPedidos atendidos na ordem dos timestamps logicos
Complexidade2(N-1) mensagens por entrada na SC
Channel como semaforo: O replyChan e um channel Go que age como contador. A goroutine chamadora fica bloqueada em <-ra.replyChan até receber N-1 sinais — muito mais limpo que condition variables em Java/Python.

05. Eleição de Líder — Algoritmo Bully

Eleição automatica quando o líder falha — o processo com maior ID vivo sempre vence.

Conceito Teorico

O nome "Bully" (valentao) vem do fato de que o processo com maior ID sempre "impoe" sua liderança sobre os menores. Cada processo tem um ID único e o líder e sempre o de maior ID entre os vivos.

Tipos de Mensagem

MensagemSignificado
ELECTION"Estou iniciando uma eleição"
OK"Estou vivo e tenho ID maior que você"
COORDINATOR"Eu sou o novo líder"
HEARTBEATLíder envia periodicamente para provar que esta vivo

Fluxo do Algoritmo

1Detecta falha do líder
→
2Envia ELECTION para IDs maiores
→
3Aguarda OK ou timeout
→
4Declara COORDINATOR
Diagrama de Sequencia — Eleição após falha do líder
P1        P2        P3        P4        P5(líder)
 |         |         |         |         |
 |         |         |         |         X (falha!)
 |         |         |         |
 |         |         | detecta |
 |         |         | timeout |
 |         |         |--ELEC-->P4
 |         |         |--ELEC-->P5 (sem resposta)
 |         |         |         |
 |         |         |<--OK----| (P4 vivo, ID maior)
 |         |         |         |
 |         |         |         |--ELEC-->P5 (sem resposta)
 |         |         |         |
 |         |         |         | timeout!
 |         |         |         |
 |<--------+---------+-COORD---| P4 e o novo líder!
 |         |         |         |
 | líder=4 | líder=4 | líder=4 | (heartbeats comecam)
Recuperacao — P5 volta e retoma liderança
[P5 se recupera e inicia]

P5 envia ELECTION para... ninguem (não ha ID > 5)
Ninguem responde (timeout) => P5 se declara líder
P5 envia COORDINATOR(5) para P1, P2, P3, P4

Todos atualizam: líder = P5 (o maior voltou!)

Implementação — internal/election/bully.go

bully.go — Constantes e estrutura
const (
    HeartbeatInterval = 2 * time.Second  // intervalo entre heartbeats
    HeartbeatTimeout  = 6 * time.Second  // timeout para considerar líder morto
    ElectionTimeout   = 3 * time.Second  // timeout para aguardar OK
)

type BullyElection struct {
    mu                 sync.Mutex
    nodeID             int
    totalNodes         int
    clock              *clock.LamportClock
    transport          network.Transporter
    leaderID           int
    electionInProgress bool
    lastHeartbeat      time.Time
    okReceived         chan struct{}
}
bully.go — StartElection
func (be *BullyElection) StartElection() {
    // Evita eleicoes simultaneas
    be.mu.Lock()
    if be.electionInProgress { be.mu.Unlock(); return }
    be.electionInProgress = true
    be.mu.Unlock()

    // Envia ELECTION para todos com ID maior
    sentToHigher := false
    for id := be.nodeID + 1; id <= be.totalNodes; id++ {
        err := be.transport.Send(id, network.Message{
            Type: "ELECTION", SenderID: be.nodeID,
        })
        if err == nil { sentToHigher = true }
    }

    if !sentToHigher {
        be.declareVictory()  // ninguem vivo acima -> eu sou líder
        return
    }

    // Espera OK ou timeout
    select {
    case <-be.okReceived:
        // alguem maior vai assumir, aguardo COORDINATOR
        be.mu.Lock()
        be.electionInProgress = false
        be.mu.Unlock()
    case <-time.After(ElectionTimeout):
        be.declareVictory()  // timeout, eu sou líder
    }
}
bully.go — MonitorLeader (deteccao de falha)
func (be *BullyElection) MonitorLeader() {
    for {
        time.Sleep(HeartbeatInterval)

        be.mu.Lock()
        isLeader := be.leaderID == be.nodeID
        elapsed := time.Since(be.lastHeartbeat)
        inElection := be.electionInProgress
        be.mu.Unlock()

        if !isLeader && !inElection && elapsed > HeartbeatTimeout {
            go be.StartElection()  // líder caiu!
        }
    }
}
select + time.After: O padrão Go select { case <-ch: ... case <-time.After(d): ... } modela perfeitamente o timeout do Bully — espera OK ou expira, o que vier primeiro.

06. Camada de Rede — TCP/JSON

Comunicação real entre processos via sockets TCP com mensagens JSON.

Formato da Mensagem

message.go
type Message struct {
    Type      string `json:"type"`            // "REQUEST", "REPLY", "ELECTION", etc.
    SenderID  int    `json:"sender_id"`       // ID do remetente
    Timestamp int    `json:"timestamp"`       // Timestamp lógico de Lamport
    Data      string `json:"data,omitempty"`  // Dados extras (opcional)
}

Protocolo de Mensagens

TipoAlgoritmoDescrição
REQUESTRicart-AgrawalaPedido para entrar na seção crítica
REPLYRicart-AgrawalaPermissão concedida
ELECTIONBullyInício de eleição
OKBully"Estou vivo e tenho ID maior"
COORDINATORBully"Eu sou o novo líder"
HEARTBEATBullySinal de vida periodico do líder

Transport — Envio e Recepcao

transport.go — Send e Broadcast
// Send envia mensagem para um nó especifico
func (t *Transport) Send(targetID int, msg Message) error {
    addr := fmt.Sprintf("node%d:5000", targetID)
    conn, err := net.DialTimeout("tcp", addr, 2*time.Second)
    if err != nil {
        return err  // no inacessivel (pode ter caido)
    }
    defer conn.Close()
    return json.NewEncoder(conn).Encode(msg)
}

// Broadcast envia para todos os outros nos em paralelo
func (t *Transport) Broadcast(msg Message) {
    for id := 1; id <= t.totalNodes; id++ {
        if id != t.nodeID {
            targetID := id
            go func() {
                t.Send(targetID, msg)
            }()
        }
    }
}
transport.go — Servidor TCP + handler
// StartServer inicia servidor TCP com goroutine por conexão
func (t *Transport) StartServer() {
    listener, _ := net.Listen("tcp", fmt.Sprintf(":%d", t.port))
    go func() {
        for {
            conn, _ := listener.Accept()
            go t.handleConnection(conn) // goroutine por conexão
        }
    }()
}

// handleConnection decodifica JSON e despacha para handler
func (t *Transport) handleConnection(conn net.Conn) {
    defer conn.Close()
    var msg Message
    json.NewDecoder(conn).Decode(&msg)
    if handler, ok := t.handlers[msg.Type]; ok {
        handler(msg) // despacha: "REQUEST" -> handleRequest, etc.
    }
}

// On registra handler por tipo de mensagem
func (t *Transport) On(msgType string, handler MessageHandler) {
    t.handlers[msgType] = handler
}

cmd/node/main.go
func main() {
    cfg := config.LoadConfig()  // NODE_ID e TOTAL_NODES via env

    // 1. Camada de rede
    transport := network.NewTransport(cfg.NodeID, cfg.Port, cfg.TotalNodes)
    transport.StartServer()
    time.Sleep(5 * time.Second)  // aguarda containers Docker

    // 2. Relógio lógico
    clk := clock.NewLamportClock()

    // 3. Exclusão Mútua
    ra := mutex.NewRicartAgrawala(cfg.NodeID, cfg.TotalNodes, clk, transport)

    // 4. Eleição de Líder
    bully := election.NewBullyElection(cfg.NodeID, cfg.TotalNodes, clk, transport)
    go bully.MonitorLeader()

    if cfg.NodeID == cfg.TotalNodes {
        go bully.StartElection()  // maior ID inicia como líder
    }

    // 5. Loop: solicita seção crítica periodicamente
    for {
        time.Sleep(intervaloAleatorio)
        clk.Tick()          // evento local
        ra.RequestCS()      // bloqueia até entrar
        criticalWork(...)  // simula acesso ao recurso
        ra.ReleaseCS()      // libera para outros
    }
}
